# Управление состоянием

В терминале одновременно используются NgRx store/effects, RxJS-сервисы, signals, local storage brokers, remote storage brokers и events bus. Это нормально для real-time UI, но каждый механизм должен оставаться в своей зоне ответственности.

## Общая матрица выбора

| Механизм | Когда использовать | Когда не использовать |
| --- | --- | --- |
| NgRx store/effects | Для разделяемого feature-state, который читают несколько компонентов, сервисов или виджетов; для состояния с событиями, derived selectors, init/save/reset эффектами и связью с внешними источниками. Примеры: desktop dashboard, widget settings, widget local state, terminal settings, portfolios. | Для локального состояния одного компонента, временных значений формы, UI-флагов диалога, единичных HTTP/WebSocket запросов без shared state. |
| Core RxJS service | Для потоков данных, команд, orchestration и публичного API feature. Сервис должен скрывать NgRx actions/selectors, WebSocket/HTTP детали или сложную RxJS-композицию от компонентов. | Для долгоживущего разделяемого состояния, если нужны явные события изменения, selectors, effects и предсказуемые обновления. В таком случае используй NgRx и сделай сервис-фасад. |
| Signals | Для локального component state и производных значений в template: видимость панели, выбранная вкладка, loading flag, размеры DOM, computed view model. | Для состояния, которое должно переживать уничтожение компонента, синхронизироваться между виджетами или сохраняться в storage. |
| LocalStorageService | Для простых локальных значений пользователя или технических флагов, которые не требуют NgRx lifecycle. Для widget-local-state используй `WidgetLocalStateService`, а не прямую запись в `localStorage`. | Для desktop настроек, которые должны синхронизироваться через backend storage; для live state; для обмена событиями между компонентами. |
| RemoteStorageService | Для server-side хранения пользовательских настроек или состояния, которое должно быть доступно пользователю после входа с другого устройства. Обычно используется через app-specific broker, а не напрямую из компонента. | Для mobile-only локальных настроек, ephemeral UI state, real-time данных и frequently updated market data. |
| Settings brokers | Для связывания store-сервисов с конкретным хранилищем приложения: desktop brokers читают/пишут `RemoteStorageService`, mobile/admin brokers используют `LocalStorageService`. Brokers запускаются на уровне приложения и не должны становиться источником бизнес-состояния. | Из компонентов и виджетов. Компоненты должны работать через `TerminalSettingsService`, `WidgetSettingsService`, dashboard services и другие core API. |
| EventsBusService | Для точечных cross-widget/component событий без владения состоянием: например передать выбранную цену из стакана в форму заявки. Используй `replayLast`, только если позднему подписчику нужно последнее событие. | Для настроек, persisted state, запросов к API, команд с обязательным результатом, глобального state или сценариев, где нужен порядок/история/отмена операций. |

## Практические правила

- Если состояние нужно сохранить как пользовательские настройки терминала или виджета, используй `TerminalSettingsService` или `WidgetSettingsService`; storage выбирает broker приложения.
- Если состояние относится к конкретному экземпляру виджета, не является настройкой и может быть восстановлено локально, используй `WidgetLocalStateService`.
- Если состояние разделяется между feature-частями и имеет init/update/remove/reset lifecycle, добавляй NgRx feature в `store` и отдавай наружу сервис-фасад.
- Если данные приходят из WebSocket/HTTP и не должны сохраняться как пользовательское состояние, держи их в RxJS service и отдавай наружу `Observable`.
- Если компоненту нужно только управлять локальной UI-деталью, используй `signal()`/`computed()` внутри компонента.
- Если виджеты должны слабосвязанно сообщить о пользовательском действии, публикуй событие через `EventsBusService`; payload должен быть минимальным и типизированным.
- Не обращайся к NgRx store напрямую из виджета, если уже есть core-service API для этой feature.
- Не создавай новый broker, пока не понятно, какое конкретное приложение и какое хранилище он связывает.

## Состояние виджета

Виджет должен выбирать самый узкий механизм состояния, который покрывает задачу.

| Задача | Что использовать |
| --- | --- |
| Настройки, которые пользователь меняет и ожидает увидеть после перезапуска или на другом desktop-устройстве | `WidgetSettingsService` и settings type виджета. Не пиши такие данные напрямую в `LocalStorageService` или `RemoteStorageService`. |
| Локальное состояние конкретного экземпляра виджета, которое не является настройкой, но может быть восстановлено после reload | `WidgetLocalStateService` с ключом записи и `widgetGuid`. |
| Текущие данные из HTTP/WebSocket, фильтры потока, table params, loading state потока | RxJS service или локальные `Observable`/`Subject` с корректным teardown. |
| Сложная локальная коллекция внутри виджета или utility-класса | `ComponentStore` или отдельный локальный manager, если обычных `BehaviorSubject` и чистых функций уже недостаточно. |
| UI-флаг внутри компонента: открыта панель, выбран tab, идет локальная загрузка, вычисляемая view model | `signal()` и `computed()` внутри компонента. |
| Слабосвязанное событие между виджетами, например выбор цены в одном виджете и реакция формы заявки в другом | `EventsBusService` с типизированным event key и минимальным payload. |
| Разделяемое состояние, которое нужно нескольким виджетам или приложениям | Перенеси API состояния в `terminal-core-lib`; виджет должен зависеть от core service или injection token. |

Не создавай отдельный NgRx feature-store внутри `terminal-widgets-lib` для состояния одного виджета. Если состояние действительно общее и требует NgRx lifecycle, оно должно жить в `terminal-core-lib` или в приложении-владельце, а виджеты должны работать через публичный сервис.

Не используй events bus как хранилище. Он подходит для одноразовых пользовательских событий; настройки, выбранные инструменты, портфели, списки и восстановимое состояние должны идти через settings/dashboard/local-state сервисы.

## Граница widget settings

Settings должны описывать только устойчивые пользовательские предпочтения: выбранный инструмент или портфель, режим отображения, глубину, включенные колонки, дефолтные параметры формы.

Не сохраняй в settings live market data, результаты HTTP-запросов, loading/error flags, размеры DOM, открытость локального dropdown/dialog или временный текст поиска.
